# 面向 CCF-C 导向的研究设计汇报稿

## 一、先用一句大白话说清楚这篇论文现在要做什么

这篇论文现在不准备写成：

- “我复现了一个图遗忘算法”
- “我做了几个实验，指标还可以”

而是准备写成：

> **在真实金融风控图里，如果有人要求删除数据，模型到底是真的忘掉了，还是只是看起来像忘掉了。**

更直白一点说，就是：

> **我想研究图遗忘到底是真删除，还是假删除。**

这里的“真删除”指的是：

1. 模型任务上看起来像删掉了
2. 隐私风险上也真的降下来了

这里的“假删除”指的是：

1. 任务指标看着还行
2. 但模型其实还残留着被删除对象的信息
3. 或者攻击之所以失败，只是因为模型本身已经分不清了，不是真的安全了

---

## 二、为什么要这样改，而不是继续写“统一框架”

如果按 CCF-C 的标准来写，老师最关心的通常不是“你有没有把东西做全”，而是：

1. 这个问题是不是真实的
2. 这个问题有没有研究价值
3. 你做的实验是不是能把结论讲清楚
4. 你的结论是不是不只是“某个数变好了”

现在图遗忘这个方向，已经有人做过很多：

1. 图遗忘方法
2. 节点、边、特征删除
3. benchmark 和框架

所以如果我们还说：

> “我要做一个统一图遗忘框架”

这个表述就容易显得不新。

但我们现在手里的代码和实验现象，其实更适合讲一个更有意思的问题：

> **图遗忘到底有没有真的降低隐私泄露风险。**

这个问题更贴近实际，也更容易写出自己的东西。

---

## 三、这篇论文最核心的问题是什么

最核心的问题其实就一句话：

> **删了，模型是不是真的忘了？**

把它展开，就是三个层面：

### 1. 删完以后，模型还准不准

也就是任务性能还能不能保持。

### 2. 删完以后，是不是真的比重训练快

如果删一次比从头再训一遍还麻烦，那工程上就没意义。

### 3. 删完以后，隐私风险是不是真的降了

这个最关键。因为很多时候模型在任务层面看起来像删掉了，但从攻击角度看，信息可能还在。

所以这篇论文最终不是只回答：

> “图遗忘有没有用”

而是回答：

> **图遗忘什么时候真的有用，什么时候只是表面上有用。**

---

## 四、为什么我们现在的方向比“单纯复现方法”更合适

因为当前代码其实已经给了我们一个很好的切入口。

现在已经有的基础包括：

1. `DGraphFin` 已经接进来了
2. 节点遗忘已经基本跑通了
3. 边遗忘脚本已经有了
4. 特征遗忘也已经有主脚本基础
5. 已经补了 `AUC / F1 / precision / recall`
6. 已经接了一个比原来更强的 MIA 攻击

而且更重要的是，我们已经发现了一个很有价值的现象：

> **攻击失败，不一定代表安全了。**

比如 `Cora + SGC` 上，MIA 接近随机，很可能不是因为遗忘做得好，而是因为模型本身已经过平滑了，成员和非成员本来就很难区分。

这个发现其实比“我又多跑了一个实验”有价值多了。

---

## 五、论文现在最适合怎么讲

我建议整篇论文都围绕一句话讲：

> **功能遗忘不一定等于安全遗忘。**

也就是说：

1. 模型功能上看似删掉了
2. 不代表隐私上就真的安全了

然后把这个问题分成三类删除对象来研究：

### 1. 节点遗忘

对应删一个人、删一个账号、删一个样本。

### 2. 边遗忘

对应删一条关系、一条交易链、一条关联边。

### 3. 特征遗忘

对应删一个字段、删一个敏感属性、做属性脱敏。

这样老师就很容易理解：

> 这不是在补实验，而是在研究三种真实合规场景下，模型到底能不能真正忘干净。

## 六、为什么主数据集应该放在 DGraphFin，而不是 Cora

这个也可以直接跟老师说明白。

`Cora` 这种数据集的问题是：

1. 太像教学数据集
2. 图结构太同质
3. 用 `SGC` 这类线性化模型容易过平滑
4. 攻击接近随机时，很难判断到底是模型安全了，还是模型本来就不可分

而 `DGraphFin` 的优点是：

1. 更接近真实金融风控图
2. 更符合隐私合规场景
3. 类别不平衡、图结构复杂，更像真实业务
4. 更容易讲“安全收益是不是成立”这个故事

所以最合适的安排是：

1. `DGraphFin` 做主结果
2. `Cora + SGC` 只拿来说明“伪安全”现象
3. `Elliptic++` 后面有时间再做扩展验证

---

## 七、最推荐的实验安排

下面这个安排，是比较符合当前代码基础，也最像论文主线的做法。

### 1. 第一条主线：节点遗忘

这是现在最成熟、最容易先做稳的一条。

要回答三个问题：

1. 删节点以后，模型性能还能不能接近重训练
2. 删节点是不是明显比重训练快
3. 删节点以后，成员推断攻击有没有真的下降

怎么做：

1. 用 `DGraphFin`
2. 优先用 `binary` 模式
3. 跑 `Before / Unlearning / Retrain`
4. 设几个删除规模，比如 `50 / 100 / 500 / 1000`
5. 做两种删法：
   随机删节点
   高影响节点删除

为什么这样做：

1. 能先把最稳的一条线做出来
2. 能直接回答“有没有用、快不快、安不安全”
3. 这部分最适合先写成正文主结果

### 2. 第二条主线：伪安全诊断

这个部分很重要，甚至可能是全文最有亮点的一部分。

要回答的问题是：

> 攻击接近随机，到底是因为遗忘成功了，还是因为模型本来就学塌了。

怎么做：

1. 拿 `Cora + SGC` 和 `DGraphFin + binary` 做对比
2. 比较：
   下游 AUC / F1
   MIA AUC
   posterior 分布
   预测是不是塌到单类

为什么这样做：

1. 可以把“攻击失败”分成真失败和假失败
2. 这部分能明显提高论文深度
3. 它不是简单补实验，而是在做机制分析

### 3. 第三条主线：边遗忘

这一条是为了让论文更像“图隐私”而不是“样本删除”。

要回答的问题：

1. 删边以后任务性能怎么变
2. 删边是不是比重训练快
3. 被删关系还能不能被恢复出来

怎么做：

1. 在 `DGraphFin` 上做
2. 先做随机删边
3. 再做关键边删除
4. 对比 `Before / Unlearning / Retrain`

为什么这样做：

1. 边是图结构最核心的对象
2. 只做节点容易让论文太像普通删除
3. 边遗忘更能体现图结构残留问题

### 4. 第四条主线：特征遗忘

这一条最贴近“字段删除”的合规需求。

要回答的问题：

1. 删敏感特征以后任务损失大不大
2. 特征遗忘是不是比删节点更温和
3. 被删掉的属性还能不能被推断出来

怎么做：

1. 先做随机特征删除
2. 再做重要特征删除
3. 先统一用 MIA 做弱审计
4. 后面有精力再补属性推断攻击

为什么这样做：

1. 更贴近现实中的“删字段”
2. 更容易体现应用价值
3. 论文会更完整

## 八、老师最容易听明白的一版“论文价值”

如果要用最直接的大白话去讲，我建议这么说：

> 现在很多图遗忘论文，重点放在“删完以后模型还能不能继续用”。但我这篇论文更想进一步看“删完以后模型是不是真的不记得了”。因为从合规角度看，模型还能不能用是一回事，隐私是不是还在泄露是另一回事。我的思路就是把删除请求分成删节点、删边、删特征三类，然后分别去看它们在性能、效率和隐私风险上的变化，重点分析图遗忘到底是在真实地消除信息，还是只是在表面上完成了删除。

这段话的好处是：

1. 很容易懂
2. 不会把题目说得太散
3. 能直接体现你和已有工作的区别

## 九、为什么这更符合 CCF-C 风格

CCF-C 风格的论文一般不一定要求你有特别重的理论创新，但很看重：

1. 问题是不是现实
2. 实验是不是系统
3. 分析是不是到位
4. 结论是不是有启发

按这个标准看，我们现在这个版本比“统一框架”更合适，因为它具备：

### 1. 明确场景

金融风控图、反欺诈图、隐私删除请求。

### 2. 明确问题

图遗忘到底有没有真正降低隐私风险。

### 3. 明确实验线

节点、边、特征三类删除对象。

### 4. 明确分析重点

区分“功能遗忘”和“安全遗忘”。

### 5. 明确应用启发

哪些删除能真正降低风险，哪些删除只是看起来删掉了。

## 十、接下来最稳妥的推进顺序

如果要跟老师汇报下一步怎么做，可以按下面这个顺序说。

### 第一阶段

先把 `DGraphFin` 上的节点遗忘主线做稳：

1. 固定 `binary` 配置
2. 跑多个删除规模
3. 跑 `Before / Unlearning / Retrain`
4. 把 MIA 协议先稳定下来

### 第二阶段

补“伪安全”诊断：

1. 对比 `Cora + SGC` 和 `DGraphFin`
2. 解释为什么攻击失败不一定等于安全成功

### 第三阶段

推进边遗忘：

1. 随机删边
2. 关键边删除
3. 看关系是否仍可恢复

### 第四阶段

推进特征遗忘：

1. 随机特征删除
2. 重要特征删除
3. 条件允许时补属性推断攻击

### 第五阶段

如果时间允许，再加 `Elliptic++` 做扩展验证。

## 十一、给老师口头汇报时可直接使用的版本

下面这段可以直接拿去说。

“我现在准备把论文方向重新收紧一下，不再把它写成单纯的图遗忘算法复现或者统一框架，而是想做成一个面向金融风控图的图遗忘安全审计研究。更具体地说，我想回答的是：数据删掉之后，模型到底是真的忘了，还是只是任务上看起来像忘了。因为从合规角度看，模型还能不能继续用是一回事，隐私是不是还在泄露是另一回事。

我现在的思路是把删除请求分成三类：节点遗忘、边遗忘和特征遗忘，分别对应删人、删关系和删字段。然后对每一类都从三个角度评估：第一是性能还能不能保持，第二是是不是比完全重训练更快，第三是隐私风险是不是真的降下来了。

当前我已经把 DGraphFin 接进主代码，节点遗忘链路基本跑通，也补了 AUC、F1 和 MIA 等指标。接下来我准备先把 DGraphFin 上的节点遗忘结果做稳，再把边遗忘和特征遗忘推进起来。论文里我还想重点分析一个问题，就是攻击失败不一定等于隐私安全，因为像 Cora 加 SGC 这种设置里，攻击接近随机可能只是因为模型本身过平滑了，不是真的忘干净了。所以我希望最后这篇论文不只是说图遗忘能不能跑，而是更进一步说明图遗忘什么时候真的安全，什么时候只是表面上删掉了。”

## 十二、最后一句总结

现在这篇论文最合适的说法不是：

> “我做了一个统一图遗忘框架。”

而应该是：

> **我在真实风控图上研究图遗忘到底能不能真正降低隐私风险，并分析它什么时候是真遗忘，什么时候只是看起来忘了。**
