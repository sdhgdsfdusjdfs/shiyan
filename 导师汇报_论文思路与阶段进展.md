# 面向导师的阶段汇报：论文思路、研究问题、实验设计与推进方案

## 一、当前论文应如何重新定位

结合现有代码、已完成实验和已有文献进展，这篇论文不宜再定位为：

- “我复现了一个图遗忘方法”
- “我搭了一个统一图遗忘框架”
- “我做了节点、边、特征三类删除，所以论文就完整了”

更稳妥、也更符合当前基础的定位应当是：

> **面向真实风控/反欺诈图场景，系统研究图遗忘在节点、边、特征三类删除请求下的效用、效率与隐私风险表现，重点回答“功能遗忘是否真的等于安全遗忘”这一问题。**

进一步收紧后，本文最核心的论文主张应当是：

> **本文不是强调提出一个全新的图遗忘理论方法，而是面向金融风控图，系统研究图遗忘在不同删除对象上的真实安全收益，并分析哪些情况下图遗忘只实现了功能删除，而未实现安全删除。**

这个定位有三个直接好处：

1. 避开“统一 benchmark / 统一框架”这类已有工作密集的表述。
2. 更贴合当前仓库已经具备的代码与实验能力。
3. 即使结果偏负，也仍然有清晰的论文价值，因为可以形成“图遗忘安全边界”的实证结论。

---

## 二、为什么要这样改论文主线

现有图遗忘方向已经不是空白问题。已有工作已经做过：

1. 图遗忘算法本身。
2. 通用图遗忘框架。
3. 节点/边/特征删除支持。
4. benchmark 或系统性评估。

因此，如果继续把论文写成：

> “在图上做遗忘，然后报效用和效率，再补一个 MIA”

那么创新度会明显不足，尤其不足以自然支撑 CCF-C 导向的论文写法。

但从当前代码和实验现状来看，我们有一个更有价值、也更贴近实际的切入点：

1. 现有 `DGraphFin` 节点遗忘链路已经跑通。
2. 已经补齐了 `AUC / F1 / precision / recall` 等比 Accuracy 更可靠的指标。
3. 已经接入了基于 posterior-shadow 的 MIA，而不是最初的弱攻击基线。
4. 已经观察到 `Cora + SGC` 中 **over-smoothing** 会让攻击接近随机，从而产生“伪安全”现象。

这说明当前最值得做的，不是继续泛化论文标题，而是把论文做实为：

> **图遗忘安全审计与机制分析**

也就是不仅看：

- 遗忘后模型还准不准
- 遗忘后模型快不快

更要看：

- 遗忘后隐私攻击是不是真的更难了
- 如果攻击失败，这种失败到底来自遗忘成功，还是来自模型本身已经不可分

## 三、论文真正要回答的研究问题

相比“某个遗忘算法效果如何”，本文更适合围绕以下四个层次展开。

### 1. 功能层问题

在删除节点、边或特征后，图遗忘能否在不完全重训练的前提下维持可接受的任务性能。

### 2. 代价层问题

图遗忘相比完全重训练，是否具有明显的时间和计算优势。

### 3. 安全层问题

图遗忘后，成员、关系或属性泄露风险是否真的下降。

### 4. 机制层问题

如果安全风险下降，原因是什么；如果没有下降，信息又是如何残留下来的。

因此，本文最终要回答的不是简单的“遗忘有没有用”，而是：

> **图遗忘在什么条件下既实现了功能删除，又实现了安全删除；又在什么条件下只完成了功能层面的删除近似，而没有真正降低隐私泄露风险。**

---

## 四、本文最推荐的总体叙事

整篇论文建议围绕一句话组织：

> **功能遗忘不一定等于安全遗忘。**

在这个总叙事下，三类删除对象的角色分别是：

1. **节点遗忘**
   对应个体级删除请求，例如账户注销、用户移除、样本级彻底删除。

2. **边遗忘**
   对应关系级删除请求，例如交易链路删除、社交关系撤销、敏感交互断开。

3. **特征遗忘**
   对应属性级删除请求，例如敏感字段删除、画像脱敏、属性级合规处理。

这三部分不是三个孤立实验，而是三种不同粒度的隐私删除场景：

> **个体级删除、关系级删除、属性级删除**

也就是说，本文不再强调“统一 benchmark”，而强调：

> **面向真实合规场景的三类删除对象安全审计**

---

## 五、为什么当前应优先以 DGraphFin 为核心

当前代码和已有结果已经很清楚地说明：`Cora + SGC` 适合调试，不适合作为最终主结果。

主要原因如下：

1. `Cora` 是高度同质的引文图。
2. `SGC` 线性化之后更容易出现过平滑。
3. 一旦后验分布过于相似，posterior-based MIA 很容易接近随机。
4. 此时“攻击失败”不能直接被解释为“隐私保护成功”。

而 `DGraphFin` 更适合作为主实验数据集，因为：

1. 更贴近真实金融风控/反欺诈业务。
2. 类别不平衡更明显，更接近现实困难。
3. 图结构复杂，能更真实地体现结构残留问题。
4. 更适合讲“隐私合规”而不仅仅是“方法跑通”。

因此论文的数据集策略建议是：

1. `DGraphFin` 作为主数据集。
2. `Cora + SGC` 仅作为负面现象或局限性分析。
3. `Elliptic++` 作为时间允许时的扩展验证集。

### 3. MIA 攻击已经升级

`utils.py` 中的 `membership_inference_attack` 已经不是最初的轻量攻击，而是：

1. 使用 shadow model。

3. 输出更详细的诊断日志。

### 4. 当前最成熟的实验路径

当前最成熟、最适合先做稳的实验路径是：

> **DGraphFin + binary mode + node unlearning + compare retrain + posterior-shadow MIA**

这应当作为全文第一主实验线。

## 七、论文最核心的创新落点应放在哪里

考虑已有工作和现有基础，本文最值得强调的创新不应是“提出全新算法”，而应放在以下四点。

### 1. 真实场景导向

不是在玩具引文图上做图遗忘，而是在金融风控图上讨论隐私删除与安全风险。

### 2. 对象对齐的隐私审计

删除对象与攻击目标一一对应：

1. 节点删除 -> 成员推断攻击
2. 边删除 -> 链接推断 / 关系泄露攻击
3. 特征删除 -> 属性推断攻击

### 3. 功能删除与安全删除的区分

不再默认“遗忘后攻击变差 = 隐私已安全”，而是进一步分析：

1. 是不是模型本身已经不可分
2. 是不是过平滑让后验天然相似
3. 是不是攻击方向反了
4. 是不是类不平衡导致阈值失真

### 4. 机制分析而不仅仅是报数

最终结论不只是“某个数值升了或降了”，而是：

1. 哪类删除更容易留下结构残留
2. 哪类安全收益更依赖于删除对象的重要性
3. 哪些攻击失败是“真安全”
4. 哪些攻击失败其实是“伪安全”

---

## 八、实验主线一：节点遗忘

节点遗忘是当前最成熟、最值得优先固化的实验线。

### 1. 为什么做

1. 最贴近“用户删除账户/要求彻底删除其数据”的现实合规场景。
2. 当前代码基础最成熟。
3. 最适合作为整篇论文的主结果部分。

### 2. 要回答的问题

1. 节点遗忘后的任务性能能否接近完全重训练。
2. 节点遗忘是否显著快于完全重训练。
3. 节点遗忘后成员推断风险是否真的下降。
4. 如果 MIA 不下降，是模型仍记住了节点，还是攻击协议尚不充分。

### 3. 实验设计

#### （1）数据集

1. 主数据集：`DGraphFin`
2. 扩展数据集：`Elliptic++`（时间允许时加入）

#### （2）训练配置

优先固定为：

1. `binary mode`
2. 合理使用 `balance_train`
3. 保持当前较稳定的超参数组合

原因是：

1. 当前 `OVR` 容易在不平衡条件下坍塌。
2. `binary` 至少已经表现出较稳定的排序能力。

#### （3）删除规模

建议至少三档：

1. 小规模：`50 / 100`
2. 中规模：`500`
3. 大规模：`1000`

#### （4）删除策略

建议做两类：

1. **随机删节点**
2. **高影响节点删除**

高影响节点可以用以下方式近似定义：

1. 度数较高的节点
2. 风险类节点
3. 模型置信度较高的节点
4. 对训练损失贡献较大的节点

#### （5）对比对象

至少包含：

1. 原始模型（Before Unlearning）
2. 遗忘模型（Unlearning）
3. 完全重训练模型（Retrain）

### 4. 指标设计

#### 效用指标

1. `AUC`
2. `F1`
3. `Precision`
4. `Recall`
5. 必要时附 `Accuracy`

#### 效率指标

1. 遗忘更新时间
2. 完全重训练时间
3. 相对加速比

#### 安全指标

1. `MIA AUC Before`
2. `MIA AUC After`
3. `MIA AUC Retrain`

### 5. 理想结论

理想情况下，这部分可得出：

1. 节点遗忘在合理删除规模下接近重训练性能。
2. 节点遗忘显著快于完全重训练。
3. 节点遗忘后成员推断攻击被削弱。

如果第三点不成立，也仍然能形成有价值的结论：

1. 节点遗忘的任务效果接近重训练。
2. 但隐私收益有限。
3. 图结构传播残留使得个体级删除不自动等价于成员泄露消除。

---

## 九、实验主线二：节点遗忘中的“伪安全”诊断

这是本文最值得写深的一条机制分析线。

### 1. 为什么做

当前已经观察到：

1. `Cora + SGC` 中 MIA AUC 接近 `0.5`
2. 这未必说明隐私保护成功
3. 更可能是过平滑导致成员与非成员天然难分

这个现象本身非常有论文价值，因为它说明：

> **攻击失败不一定等于遗忘成功。**

### 2. 要回答的问题

1. 攻击接近随机，到底来自遗忘成功还是模型不可分。
2. 风控图与引文图在这一点上是否存在显著差异。
3. 过平滑、类不平衡、阈值偏差如何共同影响隐私评估。

### 3. 实验设计

对比两组设置：

1. `Cora + SGC`
2. `DGraphFin + binary`

在每组下观察：

1. 下游 AUC / F1
2. MIA AUC
3. 成员与非成员 posterior 分布差异
4. 预测类别分布是否坍塌

### 4. 论文价值

这一部分可以帮助论文写出一个很强的机制结论：

> **图遗忘后的“安全收益”必须建立在目标模型本身保留了可分隐私信号这一前提下，否则攻击失效可能只是模型表示退化，而非真正的安全删除。**

## 十、实验主线三：边遗忘

边遗忘是把论文从“样本删除”拉回“图结构隐私”的关键。

### 1. 为什么做

1. 只做节点遗忘会让论文看起来过窄。
2. 边是图学习区别于普通样本学习的核心结构单位。
3. 边删除更贴近交易链路删除、社交关系撤销等现实合规场景。

### 2. 要回答的问题

1. 删除关系后模型性能如何变化。
2. 边遗忘是否比完全重训练更高效。
3. 被删除关系是否仍可从模型中恢复或推断。
4. 边遗忘是否比节点遗忘更容易暴露结构残留问题。

### 3. 实验设计

#### （1）数据集

1. 主数据集：`DGraphFin`

#### （2）删除规模

建议三档：

1. 小规模删边
2. 中规模删边
3. 大规模删边

#### （3）删除策略

至少两类：

1. **随机删边**
2. **关键边删除**

关键边可用以下方式定义：

1. 高风险节点之间的连接
2. 跨类别关键边
3. 高频交易边
4. 高嵌入相似度边

#### （4）对比方法

1. 原始模型
2. 边遗忘模型
3. 完全重训练模型

### 4. 指标设计

#### 效用

1. `AUC`
2. `F1`

#### 效率

1. 更新时间
2. 加速比

#### 安全

建议分两阶段推进：

第一阶段：

1. 基础性能差异
2. MIA 作为弱安全审计
3. 被删边两端输出/表示差异变化

第二阶段：

1. 链接推断攻击
2. 关系可恢复性分析

### 5. 理想结论

1. 边遗忘比节点遗忘更细粒度，更接近真实结构合规操作。
2. 关键边删除的效用损失与隐私影响通常大于随机删边。
3. 边遗忘具有明显效率优势，但结构残留问题也更突出。

## 十一、实验主线四：特征遗忘

特征遗忘是最能体现“字段级合规删除”的实验线。

### 1. 为什么做

1. 现实中的很多合规请求不是删掉整个人，而是删掉某类敏感属性。
2. 特征遗忘比节点删除更细粒度。
3. 这部分最能体现论文的应用价值。

### 2. 要回答的问题

1. 删除敏感特征后任务性能损失多大。
2. 特征遗忘能否比节点删除更温和地实现合规。
3. 被删除敏感属性是否仍可被模型推断。

### 3. 实验设计

#### （1）特征选择方式

建议两类：

1. **随机特征删除**
2. **重要特征删除**

重要特征可以依据：

1. 模型权重绝对值
2. 特征重要性评分
3. 对正类预测的贡献度
4. 风险解释性

#### （2）对比对象

1. 原始模型
2. 特征遗忘模型
3. 完全重训练模型

### 4. 指标设计

#### 效用

1. `AUC`
2. `F1`

#### 效率

1. 更新时间
2. 加速比

#### 安全

分两阶段：

第一阶段：

1. 统一先用 MIA 做弱审计

第二阶段：

1. 引入属性推断攻击

### 5. 理想结论

1. 特征遗忘在保留结构的前提下完成敏感信息消除。
2. 删除关键敏感特征后，属性泄露风险更明显下降。
3. 特征遗忘相较节点遗忘对任务性能更温和，但隐私收益强依赖于被删特征的重要性。

## 十二、一个必须加入正文的实验：删除策略敏感性分析

这一部分很重要，因为它能把论文从“报均值”提升到“解释边界”。

### 1. 为什么做

现实中被删的对象并不总是随机的。高风险节点、关键边、敏感特征往往比随机对象更重要，也更可能留下残留。

### 2. 怎么做

在三类删除对象上都加入两类对比：

1. 随机删除
2. 高影响删除

### 3. 要分析什么

1. 删除对象的重要性是否影响效用损失。
2. 删除对象的重要性是否影响安全收益。
3. 哪类删除对象更容易形成“删了但没忘干净”的情况。

### 4. 论文意义

这部分能支持一个很强的结论：

> **图遗忘的效果不仅取决于删了多少，还高度取决于删掉的是什么。**

---

## 十三、一个必须加入正文的实验：训练设置影响分析

### 1. 为什么做

当前代码和结果已经显示：

1. `OVR` 在 `DGraphFin` 上容易坍塌。
2. `binary` 更稳定。
3. Accuracy 在类别极不平衡时非常容易误导。

这些都不应该只作为调参记录，而应上升为正式讨论。

### 2. 建议做的对比

1. `OVR` vs `binary`
2. `balance_train` 开 / 关
3. 不同 `num_steps`
4. 不同 `lam`

### 3. 要回答的问题

1. 为什么某些设置下 Accuracy 很高但 AUC / F1 很差。
2. 为什么模型塌缩时 MIA 会失真。
3. 为什么安全审计必须建立在目标模型具备基本判别能力的前提下。

### 4. 论文价值

这部分可以写出一个很重要的提醒：

> **隐私评估的前提是模型本身是“可学且可泄露”的；否则安全结论可能被训练塌缩和输出平滑现象误导。**

## 十五、最推荐的实验推进顺序

### 第一阶段：固化节点遗忘主线

1. 固化 `DGraphFin + binary` 稳定配置。
2. 跑通 `Before / Unlearning / Retrain`。
3. 补齐多组删除规模实验。
4. 跑随机删除与高影响删除两组。
5. 固化一组可写正文的稳定结果。

### 第二阶段：完成“伪安全”诊断

1. 对比 `Cora + SGC` 与 `DGraphFin + binary`。
