# 面向导师的阶段汇报：论文选题、技术路线与当前实验结论

## 一、目前这篇论文准备写什么

本论文拟围绕**图神经网络场景下的隐私合规遗忘问题**展开，面向风控/反欺诈图数据，研究当用户提出删除请求后，模型能否在**尽量不重训**的前提下，快速、有效地“忘掉”指定信息，并评估这种遗忘是否真的降低了隐私泄露风险。

当前计划将论文组织为一个统一的图隐私合规研究框架，覆盖三类典型遗忘请求：

1. **节点遗忘（Node Unlearning）**
   对应“账号删除”“用户被强制移除”等场景。
2. **边遗忘（Edge Unlearning）**
   对应“关系撤销”“交易关系删除”“敏感关联断开”等场景。
3. **特征遗忘（Feature Unlearning）**
   对应“敏感属性脱敏”，即保留节点但删除其某类敏感特征。

论文最终希望从三个维度来评估图遗忘方法：

1. **效用（Utility）**：遗忘后模型任务性能下降多少。
2. **效率（Efficiency）**：相比完全重训练，遗忘更新能节省多少时间和计算成本。
3. **安全性（Security）**：遗忘后隐私泄露风险是否下降，重点用成员推断攻击（MIA）等攻击进行评估。

---

## 二、论文的核心思路

### 1. 研究主线

目前最清晰的主线是：

> 在真实风控/反欺诈图场景中，构建一个面向隐私合规的图遗忘实验框架，系统评估不同遗忘对象在**效用-效率-安全性**三方面的表现，并分析图遗忘是否真的能缓解隐私泄露。

这条主线比“单独跑一个图遗忘算法”更完整，也更接近论文表达。

### 2. 为什么这样写

这样设计主要有四个原因：

1. **单纯做“图遗忘 + 一个 MIA 实验”创新性不够强。**
   现在图遗忘和 MIA 都已经是比较成熟的研究方向，如果只是在公开小数据集上复现实验，论文说服力有限。

2. **风控/反欺诈图更贴近真实应用。**
   当前代码和文档已经把重点放在 `DGraphFin` 上，这类金融风控图天然存在类别不平衡、关系异构、隐私风险强等特点，比 `Cora` 这类同质引文图更适合讲“隐私合规”故事。

3. **三类遗忘请求更符合“合规系统”而不是“单点算法”。**
   节点、边、特征分别对应现实中的删除账号、删除关系、删除敏感属性，能把论文从“方法复现”提升为“场景化隐私合规框架”。

4. **效用、效率、安全三维评估更适合毕业论文写法。**
   这样不仅能报告性能，还能解释“为什么有效”“代价是什么”“是否真的降低攻击风险”，结构更完整。

## 四、目前论文为什么优先写 DGraphFin，而不是继续围绕 Cora

当前文档已经给出比较明确的判断：`Cora + SGC` 虽然适合调代码，但不适合作为最终论文主结果。

原因是：

1. `Cora` 是高度同质的引文图，节点之间过于平滑。
2. 当前使用的 SGC/线性化框架容易出现 **over-smoothing**。
3. 一旦过平滑，成员与非成员在后验分布上几乎没有差别，MIA 就会接近随机猜测。
4. 这种现象不一定是代码错，而更可能是“模型结构 + 数据特性”的共同结果。

因此当前论文策略更偏向：

1. 把 `Cora + SGC` 作为一个**负面现象/局限性分析**。
2. 把 `DGraphFin` 作为**主实验数据集**。
3. 后续再考虑是否引入 `Elliptic++` 作为扩展验证数据集。

## 五、各项实验的初步结论

下面这部分是当前最适合向导师汇报的内容。

### 实验 1：基础框架迁移与数据接入

**实验内容**  
将原始 `sgc_unlearn` 代码从引文图设置迁移到 `DGraphFin`，修复数据加载、`mask` 对齐、指标统计和调试流程。

**当前结论**  
1. `DGraphFin` 已经成功接入当前主实验脚本。
2. 原先阻塞实验的 `val_mask/valid_mask`、数据加载方式、设备迁移等问题，代码层面已经处理。
3. 现在项目已经具备做 `DGraphFin` 节点遗忘实验的基础条件。

**意义**  
这一步说明论文已经从“看思路”进入“能做真实实验”的阶段。

### 实验 2：Cora + SGC 下的 MIA 诊断

**实验内容**  
在较经典的 `Cora + SGC` 组合上测试成员推断攻击。

**当前结论**  
1. MIA AUC 接近 `0.49 ~ 0.50`。
2. 成员/非成员后验输出几乎没有可分性。
3. 更合理的解释不是“攻击实现有 bug”，而是出现了明显的 **posterior over-smoothing**。

**论文上可以怎么写**  
这个实验更适合作为一个**局限性发现**：
线性化图模型在高度同质图上可能天然削弱后验差异，导致 posterior-based MIA 难以奏效。

**初步结论**  
`Cora + SGC` 不适合作为论文的核心安全实验平台，但可以保留为负面结果分析。

### 实验 3：DGraphFin 上的 OVR 训练实验

**实验内容**  
在 `DGraphFin` 上先沿用较通用的 OVR 训练方式，观察任务性能和 MIA 表现。

**当前结论**  
1. 某些设置下准确率可以很高，接近 `0.99`。
2. 但 `F1` 和 `AUC` 仍可能只有 `0.50` 左右。
3. 这说明模型可能出现了**多数类主导/预测坍塌**，并不是真的学到了有效判别能力。

**论文上可以怎么写**  
在反欺诈图这种强类别不平衡数据上，**Accuracy 不能作为主指标**，必须以 `AUC` 和 `F1` 为主。

**初步结论**  
`DGraphFin` 上直接用 OVR 方案不够稳，容易造成“看起来准确率高、实际上模型无效”的假象。

### 实验 4：DGraphFin 上的 Binary 模式实验

**实验内容**  
将任务从容易坍塌的 OVR 设置转到更适合当前采样条件的 `binary` 模式。

**当前结论**  
1. 下游任务 AUC 已经提升到大约 `0.69 ~ 0.70`。
2. 说明模型已经学到一定的**排序信号**，不再是完全随机。
3. 但 `F1` 仍然很低，文档中记录约为 `0.03`，且正类精度很低。
4. 当前现象更像是“会排序，但阈值决策很差”。

**论文上可以怎么写**  
可以解释为：
当前模型具备一定风险排序能力，但在极不平衡场景下，默认决策阈值无法提供理想的正类识别效果。

**初步结论**  
`binary` 模式是目前 `DGraphFin` 节点遗忘任务中最稳定的诊断路径，也是当前最值得继续推进的主线。

### 实验 5：升级后的 MIA（posterior-shadow MIA）实验

**实验内容**  
将原本较弱的轻量 MIA，升级为参考 `rebMIGraph` 思路的 posterior-shadow 攻击流程。

**当前结论**  
1. 攻击流程比原来更规范，已经不只是简单拿置信度做分类。
2. 但在当前 debug 规模的 `DGraphFin` 节点实验里，MIA 仍然常接近 `0.50`。
3. 个别记录中还出现过约 `0.41` 的结果，提示可能存在攻击方向反转或标签约定问题。
4. 当前可以确认：**安全评估链路已经升级，但最终隐私结论还不能下得太早。**

**论文上可以怎么写**  
现在更适合表述为：
“我们已经构建了较强的 MIA 评估管线，但当前实验表明，图遗忘后的隐私改善结论仍需在更稳定设置和更大规模实验中进一步验证。”

**初步结论**  
MIA 模块已经进入“可研究、可诊断”阶段，但还没有形成可以直接写成最终论断的强结果。

### 实验 6：边遗忘与特征遗忘的整体准备情况

**实验内容**  
检查代码结构和文档规划，评估后续扩展性。

**当前结论**  
1. 代码层面已经存在 `sgc_edge_unlearn.py`，说明边遗忘具备扩展基础。
2. 文档规划已明确将边遗忘作为下一阶段重点。
3. 特征遗忘在研究叙事上非常重要，因为它更符合“敏感属性删除”这一合规场景。
4. 但截至目前，这两部分还没有形成可向论文正文直接提交的稳定实验结果。

**初步结论**  
节点遗忘是当前最成熟的主实验，边遗忘和特征遗忘属于下一阶段的核心扩展方向。

## 六、当前论文最合适的写法

如果按照现在的进展，我建议论文暂时按下面这个结构写：

### 1. 问题背景

围绕图神经网络在风控/反欺诈场景中的隐私合规需求，说明现实中存在三类删除请求：

1. 用户节点删除
2. 敏感关系删除
3. 敏感属性删除

### 2. 研究目标

构建统一图遗忘实验框架，回答三个问题：

1. 遗忘后任务性能损失有多大？
2. 遗忘比完全重训练快多少？
3. 遗忘后是否真的降低了成员推断等隐私攻击风险？

### 3. 方法与实验路线

1. 以 `sgc_unlearn` 为基础搭建遗忘框架。
2. 以 `DGraphFin` 作为主数据集。
3. 先完成节点遗忘主线。
4. 用 `MIA` 评估隐私风险变化。
5. 再扩展到边遗忘、特征遗忘，以及更丰富的攻击评估。

### 4. 当前阶段性发现

1. `Cora + SGC` 中出现了明显的 over-smoothing，导致 posterior-based MIA 近乎失效。
2. `DGraphFin` 更适合作为现实风控隐私评估平台。
3. 在 `DGraphFin` 上，简单看准确率会误判模型效果，必须重点看 `AUC/F1`。
4. `binary` 模式已经证明当前模型具备一定任务判别能力。
5. 但隐私攻击结果仍不够稳定，因此论文当前更适合写“阶段性评估 + 现象分析”，而不是过早下最终结论。
